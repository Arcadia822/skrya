# 每日输出模板反馈旅程

## 目标

让用户用自然语言调整每日简报的呈现方式，同时把可复用偏好沉淀成稳定规则，避免每次生成日报时重新解释一遍同样的审美和信息密度要求。

## 默认日报模板

日报默认面向聊天窗口阅读，用轻量线框块承载每条内容。它不依赖右侧边框对齐，所以比完整制表符更适合中文、英文和链接混排：

```markdown
# YYYY-MM-DD｜主题名｜每日简报

## 事件线更新

┌─ **【事件线】事件线名称**
│ 一句说明今天发生了什么。
│
│ 后续看点：一句说明下一步该观察什么。
└

## 今日简讯

┌─ **【简讯1】事件标题**
│ 一句判断或摘要。
│ 可以继续换行补充判断。
│
│ 信源：[来源名](url) [来源名](url)
└

┌─ **【简讯2】事件标题**
│ 一句判断或摘要。
│
│ 信源：[来源名](url)
└

---

## 系统提示

- 执行时间：YYYY-MM-DD HH:MM（Asia/Shanghai）
- 执行状态：完成：生成 N 条简讯，更新 M 条事件线。
- 扫描时间范围：最近 24 小时（默认）
- 可继续操作：
  - A. 详细分析指定今日简讯，例如：`A 3 5 12`。
  - B. 创建新的事件线，例如：`B 3 4 5 持续关注`。
  - C. 调整简讯和事件线的获取策略，例如：`C 6 7 我不喜欢，如果是 xxx 不要关注`。
```

如果没有事件线更新，直接从大标题进入编号简讯。每条简讯仍然保留稳定可见编号，方便用户直接回复数字继续深挖。

## 反馈入口

用户不需要知道模板文件或配置字段。系统只识别四类自然语言反馈：

1. 内容深挖：用户回复 `A 2`、`展开第 2 条`、`2 详细说说`。
2. 事件线：用户回复 `B 2 5 继续追这个公司纠纷`、`这几条以后放一起追踪`。
3. 关注偏好：用户回复 `C 6 以后少推这种`、`这种新品小道消息不用放日报`。
4. 模板偏好：用户回复 `信源放短一点`、`每条加一句影响判断`、`日报不要太长`。

## 用户旅程

1. 用户收到默认日报，看到编号、摘要、信源和 A/B/C 反馈入口。
2. 用户用自然语言指出一个具体问题，最好能带编号，但不能要求用户必须使用编号。
3. 系统判断反馈类型：内容深挖直接生成分析；事件线进入建线确认；关注偏好进入 durable topic memory；模板偏好进入输出格式偏好确认。
4. 对于会影响未来输出的反馈，系统先复述成稳定规则，例如“以后每条信源只显示最多两个来源，完整来源按需提供”。
5. 用户确认后，系统写入对应配置或模板约束；未确认时只把它当成本次聊天建议。
6. 下一次日报应用新规则，并在必要时用一句话提示“已按你之前的偏好压缩信源展示”。

## 模板偏好规则

模板反馈只应该改变呈现方式，不应该改变事件收集范围。典型映射如下：

- “信源太长”：保留单独的信源行，但限制展示数量，完整来源继续写入运行产物。
- “我想看影响”：在每条摘要里加入一句判断，不新增二级标题。
- “太啰嗦”：压缩每条摘要，不减少真实高价值事件。
- “别用英文来源名”：优先显示中文来源名；没有来源名时使用域名。
- “来源也要展示”：默认日报每条在空行后的信源行显示 compact source references。
- “中文表格线不齐”：放弃完整制表符和 Markdown 表格，使用只有左侧边界的轻量线框块，避免右侧对齐问题。

## 成功标准

- 用户能在一次反馈内表达模板问题。
- 系统能区分“我要改日报长相”和“我要改关注范围”。
- 所有未来可复用的偏好都需要确认后再持久化。
- 大标题包含执行日期和主题。
- 简讯主体与系统提示之间有 `---` 分割线。
- 系统提示有独立标题，并包含执行时间、执行状态、扫描时间范围和 A/B/C 反馈入口。
- 日报保持统一的编号线框块，不把前几条做成特殊版式。
- 信源默认可见，但不暴露 request id、内部文件名或抓取实现细节。
